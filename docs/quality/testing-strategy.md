# 测试策略

**自我测试 = Agent 的反馈信号。** 没有测试，Agent 就没有反馈，就无法自我验证，就无法闭环。这是 AI-Driven Engineering 中最高杠杆的实践。

## 最高杠杆实践

给 Agent 验证自身工作的方法。没有验证，Agent 就没有反馈信号，你（人类）就是唯一的反馈环——这意味着 Agent 无法自我闭环。

| 没有测试 | 有测试 |
|---------|--------|
| Agent 不知道代码是否正确 | Agent 运行测试获得反馈 |
| 人类是唯一的验证者 | Agent 自己验证 |
| 每次变更都需要人工审查 | CI/CD 自动验证 |
| 开放式循环 | 自我闭环 |

## TDD + Agent

测试驱动开发与 AI-Driven Engineering 的天然结合：

1. **先写测试**：定义预期行为，如 tests.json
2. **让 Agent 实现功能**：Agent 目标明确——让测试通过
3. **提醒 Agent**："删除或修改测试是不可接受的"
4. **验证**：运行测试，全部通过 = 完成

TDD 在 AI-Driven Engineering 中不仅是开发方法论，更是 Agent 的行为约束——测试定义了"完成"的标准。

## 覆盖率 Gate：从数字到闭环

覆盖率百分比本身不是目的，但它是 Agent 获得反馈信号的基础设施。没有覆盖率 Gate，Agent 无法知道自己的代码是否被测试覆盖——这又回到了"人类是唯一验证者"的开放循环。

### 为什么不能只看总体覆盖率

```
项目总体覆盖率 75%  ← 看起来还行
    ├── 核心业务逻辑包 A: 95%  ← 很好
    ├── 核心业务逻辑包 B: 92%  ← 很好
    ├── 工具函数包 C: 30%       ← 告警！
    └── 新增代码（上周）: 45%   ← 严重问题！
```

总体数字掩盖了局部问题。覆盖率 Gate 的价值在于**按包/目录独立检查**，确保每个模块都有基本覆盖。

### 双层 Gate 的设计逻辑

- **Tier 1（防退化）**：每包/目录覆盖率 ≥ 基线 - 1.5% 容差 → 防止覆盖率在不知不觉中下降
- **Tier 2（保新增）**：变更行覆盖率 ≥ 80% → 确保新写的代码有测试

为什么是 80% 而不是 100%？因为 100% 的门槛在实践中不可行——getter/setter、纯代理函数、框架生命周期方法等天然难以单元测试。80% 是在"质量保证"和"工程可行性"之间的平衡点。

### 覆盖率 Gate 作为 Agent 的自我闭环信号

```
Agent 写代码
    ↓
Agent 运行测试 → 覆盖率 Gate 通过？
    ↓ 是                    ↓ 否
提交代码               Agent 读取失败报告
                           ↓
                      定位未覆盖的变更行
                           ↓
                      补充测试 → 重新运行
```

覆盖率 Gate 让"写测试"从人类的事后要求变成 Agent 的即时反馈。Agent 不需要人类告诉它"这里缺测试"——CI 会告诉它。

## 豁免文件策略

不是所有代码都能、都应该被单元测试。豁免不是偷懒，而是诚实地承认某些代码的测试需要不同的策略。

### 什么样的代码可以豁免

| 类型 | 示例 | 豁免原因 | 替代验证方式 |
|------|------|----------|------------|
| CLI 子进程启动 | `ExecuteStream()` 调用外部命令 | 需要 mock 整个进程 | 集成测试 |
| 框架生命周期 | Android Service `onCreate/onDestroy` | 框架控制，非业务逻辑 | 手动测试 |
| 程序入口 | `cmd/server/main.go` | 仅组装依赖 | 冒烟测试 |
| 纯 I/O 代理 | HTTP handler 调用 service 层 | 需要完整 HTTP 服务器 | API 集成测试 |

### 豁免文件的管理原则

1. **显式声明**：每个豁免文件必须在覆盖率检查脚本中列出，附带豁免原因的注释
2. **不计入 Gate**：豁免文件的覆盖率数据从 Tier 1 基线和 Tier 2 变更行中同时排除
3. **信息展示**：豁免文件在覆盖率报告中仍显示，标记为"not counted toward gate"，保持可见性
4. **定期审查**：豁免文件不是永久的——当代码重构使得单元测试可行时，应移除豁免

### 为什么不直接降低门槛

降低总体覆盖率门槛（如从 80% 降到 60%）会让所有代码的标准下降。豁免文件策略是**精确手术**——只对特定文件放行，而不是全局降标。这保持了大部分代码的高标准，同时诚实面对少数难以单元测试的场景。

## 测试分层与优先级

测试不是全有或全无。将代码按覆盖深度分类，有助于决策"下一步测什么"：

### 三级覆盖分类

| 级别 | 定义 | 行动 |
|------|------|------|
| **深覆盖** | 核心路径 + 边界条件 + 错误路径均有测试 | 维护，不退化 |
| **浅覆盖** | Happy path 有测试，边界和错误路径缺失 | 计划深化 |
| **未覆盖** | 无测试文件 | 评估是否需要覆盖 |

### 缺口优先级

**高优先级**——无测试的核心逻辑（Composable、Service、核心工具函数）：

- 这些代码的 bug 影响面最大
- Agent 最常修改这些代码
- 测试投入的 ROI 最高

**中优先级**——无测试的辅助工具（格式化函数、辅助类）：

- 功能相对简单，bug 影响面有限
- 测试编写成本低，适合"顺带"覆盖

**低优先级**——无测试的 UI 组件、配置文件、入口文件：

- UI 组件测试的 mock 成本高，ROI 低
- 优先通过手动测试和视觉回归验证

### 实际决策框架

```
这个模块需要测试吗？
    ↓
是核心业务逻辑？ → 是 → 高优先级，必须覆盖
    ↓ 否
Agent 会频繁修改？ → 是 → 高优先级
    ↓ 否
函数逻辑复杂？ → 是 → 中优先级
    ↓ 否
UI 组件？ → 低优先级，考虑视觉回归测试
    ↓ 否
纯配置/入口？ → 可豁免，集成测试覆盖
```

## 测试报告即文档

覆盖率报告不仅是 CI 的通过/失败信号，更是项目的测试状态文档。

### 报告格式

一份好的覆盖率报告应该包含：

```markdown
# 测试覆盖率报告 - 2025-05-22

## 摘要
- 总模块数: 80
- 已覆盖: 18（深覆盖: 4, 浅覆盖: 14）
- 未覆盖: 62
- 本次新增: 11 模块
- 测试结果: 448 PASS / 0 FAIL

## 本次变更
| 模块 | 变更类型 | 新增测试数 | 覆盖重点 | 状态 |
|------|---------|-----------|---------|------|
| utils/path | 新增 | 17 | 跨平台路径, 边界输入 | PASS |
| utils/format | 新增 | 21 | 格式化函数, 错误场景 | PASS |

## 覆盖缺口
### 高优先级
- useChatRender: 块解析逻辑无测试
- useChatStream: SSE 重连逻辑无测试

### 中优先级
- clipboard: 剪贴板操作无测试
- globals: 共享单例无测试

### 低优先级
- 所有 44 个 Vue 组件无测试
```

### 报告的核心价值

1. **Delta 追踪**：显示测试数量变化（`217→251 tests`），量化进度
2. **缺口登记**：明确列出仍需关注的模块，按优先级排序
3. **新增文件清单**：列出本次新增的测试文件，便于 review
4. **趋势判断**：跨报告比较，判断覆盖率是在上升还是下降

### 为什么不让 CI 输出代替报告

CI 的通过/失败是二值信号——它告诉你"行"或"不行"，但不告诉你"差多少"和"差在哪里"。覆盖率报告是**结构化的差距分析**，让 Agent 和人类都能做出有针对性的改进决策。

## 验证方法

| 方法 | 适用场景 | 示例 |
|------|----------|------|
| 测试套件 | 逻辑验证 | `pnpm test` |
| 覆盖率 Gate | 质量门控 | `./scripts/check-coverage.sh` |
| Lint/格式化 | 代码规范 | `pnpm lint` |
| Bash 命令输出 | 构建验证 | `pnpm build && echo "OK"` |
| UI 截图对比 | 视觉验证 | 截图对比原始设计 |
| 子 Agent 审查 | 代码质量 | 独立 Agent 审查变更 |

覆盖率 Gate 是新增的验证方法——它不是替代测试套件，而是将测试结果转化为可执行的门槛。

## 提示模板对比

| 模糊提示 | 精确提示 |
|---------|---------|
| "实现邮箱验证函数" | "写 validateEmail 函数。测试用例：user@example.com 为 true、invalid 为 false、user@.com 为 false。实现后运行 `pnpm test` 验证" |
| "修 bug" | "src/auth/handler.ts:47 在输入 null 时抛出 TypeError。修复并确保 `pnpm test` 通过" |
| "加个仪表盘" | "实现仪表盘设计。截图对比原始设计，列出差异并修复" |

## 测试即规范

测试不仅是质量保障，更是 Agent 理解预期行为的规格说明。测试用例告诉 Agent：
- 正常输入应该返回什么
- 边界条件应该如何处理
- 异常情况应该如何响应

**不能验证 = 不能发布。** 这是 AI-Driven Engineering 的铁律。
